# ROADMAP

불규칙한 모임의 참여 취합·정산·카풀 배정을 링크 하나로 해결하는 모임 이벤트 관리 앱 MVP 개발 로드맵입니다.

## 프로젝트 개요

**모임 이벤트 관리 앱**은 동창/스터디/친구 모임 등 비정기적·불규칙한 이벤트를 5~15명 규모의 주최자거나 참여자를 위한 서비스입니다. 주최자(Host)는 이벤트 생성·공유·카풀 배정·정산까지 한 곳에서 처리하고, 참여자(Guest)는 링크 하나로 접속해 Google 로그인 후 참여 응답과 카풀 신청을 할 수 있습니다.

핵심 가치

- **링크 하나로 참여 취합**: 이벤트 링크 공유 후 참여/불참/미정 응답을 실시간 현황으로 확인
- **자동화된 정산**: 각자 지출 기록으로 N/1 자동 계산, 복사 가능한 정산 메시지
- **간편한 카풀 배정**: 드라이버 등록 후 탑승 신청 시 배정 확정

### 기술 스택

- Next.js 15 App Router + TypeScript
- Supabase (Auth + PostgreSQL + Realtime)
- Google OAuth (소셜 로그인)
- shadcn/ui + Tailwind CSS
- React Hook Form + Zod

### 화면 구조

| 경로                      | 설명                   |
| ------------------------- | ---------------------- |
| `/`                       | 랜딩 페이지            |
| `/login`                  | Google 로그인          |
| `/dashboard`              | 내 이벤트 목록         |
| `/events/new`             | 이벤트 생성            |
| `/events/[id]`            | 이벤트 상세 (참여자용) |
| `/events/[id]/manage`     | 이벤트 관리 (주최자용) |
| `/events/[id]/settlement` | 정산                   |
| `/events/[id]/carpool`    | 카풀                   |

### 데이터베이스 테이블

`users`, `events`, `attendances`, `announcements`, `expenses`, `carpool_drivers`, `carpool_members`

## 개발 핵심 원칙 1: UI 마크업 먼저, DB·스키마는 나중에 (MUST)

> 이 원칙은 모든 Phase의 태스크 순서에 예외 없이 적용됩니다.

1. **UI/UX 마크업 선행**: 데이터가 없어도 목업(더미) 데이터로 화면(마크업)을 먼저 완성합니다.
2. **DB·스키마는 UI 확정 이후**: 화면이 확정된 이후에 Supabase 테이블·RLS를 생성합니다.
3. **단계 순서 고정**: 각 Phase는 반드시 **A) UI 마크업(더미) → B) DB 스키마(테이블·RLS) → C) 데이터 연동** 순으로 진행합니다.
4. **연동은 마지막**: 더미 데이터를 실제 Supabase 호출로 교체하는 작업은 언제나 해당 Phase의 마지막 그룹에서 진행합니다.

## 개발 핵심 원칙 2: 구현 후 반드시 테스트 (MUST)

> 이 원칙은 모든 태스크에 예외 없이 적용됩니다.

1. **구현 후 테스트는 필수**: 모든 구현 태스크는 완료 직후 반드시 테스트를 실행하며, 테스트 통과 후에만 완료(`[x]`) 처리할 수 있습니다.
2. **테스트 도구는 Playwright MCP**: 모든 기능 검증·E2E 테스트는 Playwright MCP로 진행합니다(브라우저 탐색, 요소 상호작용, 네트워크 요청 확인, 스크린샷 검증).
3. **API·비즈니스 로직은 꼼꼼히**: API 연동, 데이터 파싱/변환, N/1 계산, 상태 처리를 포함하는 태스크는 **정상 케이스 + 실패 케이스 + 엣지 케이스·경계값**을 모두 검증합니다.
4. **테스트 통과가 완료 조건**: 테스트를 통과해야만 태스크를 완료로 표시하고 다음 단계로 진행합니다.

### 테스트 검증 깊이 기준 (API·비즈니스 로직)

- **정상(Happy Path)**: 유효한 입력으로 기대 결과가 정확히 반환되는지
- **실패**: API 오류(4xx/5xx), 타임아웃, 권한 없음, 데이터 없음 상태 처리
- **엣지·경계값**: 빈 목록, 정원 초과, 0/음수/최대 인원, 만료된 이벤트, 특수문자 입력
- **데이터 정합성**: N/1 정산 금액, 참여 인원 집계값 일치 확인
- **사용자 흐름(E2E)**: 핵심 사용자 여정 전체를 처음부터 끝까지 검증

> 참고: UI 마크업(더미) 단계의 테스트는 렌더링·반응형·네비게이션 동작 검증 위주로 진행하고, 데이터 정합성·실제 검증은 데이터 연동 단계에서 집중적으로 진행합니다.

- **접근성 강화**: WCAG 2.1 AA 준수 상세 점검
