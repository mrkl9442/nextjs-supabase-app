# ROADMAP

소규모 모임의 참여 취합·정산·카풀 배정을 링크 하나로 해결하는 모임 이벤트 관리 웹 MVP의 개발 로드맵입니다.

## 프로젝트 개요

**모임 이벤트 관리 웹**은 수영/헬스/친구 모임 등 비정기 소규모 이벤트(5~15명)를 주최하거나 참여하는 일반인을 위한 서비스입니다. 주최자(Host)는 이벤트 생성·공지·카풀 배정·정산을 한 곳에서 처리하고, 참여자(Guest)는 링크 하나로 접속해 Google 로그인 후 참여 응답과 카풀 신청을 할 수 있습니다.

핵심 가치:

- **링크 하나로 끝나는 참여 취합**: 이벤트 링크 공유 → 참여/불참/미정 응답 → 실시간 현황 확인
- **자동화된 정산**: 항목별 지출 기록과 N/1 자동 계산, 복사 가능한 정산 메시지
- **간편한 카풀 배정**: 드라이버 등록 → 탑승 신청 → 배정 확정

### 기술 스택

- Next.js 15 App Router + TypeScript
- Supabase (Auth + PostgreSQL + Realtime)
- Google OAuth (이미 구현됨)
- shadcn/ui + Tailwind CSS
- React Hook Form + Zod

### 화면 구조

| 경로                      | 설명                    |
| ------------------------- | ----------------------- |
| `/`                       | 랜딩 페이지             |
| `/login`                  | Google 로그인           |
| `/dashboard`              | 내 이벤트 목록          |
| `/events/new`             | 이벤트 생성             |
| `/events/[id]`            | 이벤트 상세 (참여자 뷰) |
| `/events/[id]/manage`     | 이벤트 관리 (주최자 뷰) |
| `/events/[id]/settlement` | 정산                    |
| `/events/[id]/carpool`    | 카풀                    |

### 데이터베이스 테이블

`users`, `events`, `attendances`, `announcements`, `expenses`, `carpool_drivers`, `carpool_members`

## ⚠️ 개발 핵심 원칙 1: UI 마크업 먼저, DB·스키마는 나중에 (MUST)

> 이 원칙은 모든 Phase의 태스크 순서에 예외 없이 적용됩니다.

1. **UI/UX 마크업 선행**: 실데이터 없이 **더미 데이터**로 화면(마크업)을 먼저 완성합니다.
2. **DB·스키마는 UI 확정 후**: 화면이 확정된 뒤에 Supabase 테이블·RLS를 생성합니다.
3. **단계 순서 고정**: 각 Phase는 반드시 **A) UI 마크업(더미) → B) DB 스키마(테이블·RLS) → C) 실데이터 연동** 순으로 진행합니다.
4. **연동은 마지막**: 더미 데이터를 실제 Supabase 호출로 교체하는 작업은 항상 해당 Phase의 마지막 그룹에서 수행합니다.

## ⚠️ 개발 핵심 원칙 2: 구현 후 반드시 테스트 (MUST)

> 이 원칙은 모든 태스크에 예외 없이 적용됩니다.

1. **구현 후 테스트는 필수**: 모든 구현 태스크는 완료 직후 반드시 테스트를 수행하며, 테스트 없이 완료(`[x]`) 처리할 수 없습니다.
2. **테스트 도구는 Playwright MCP**: 모든 기능 검증·E2E 테스트는 Playwright MCP로 수행합니다 (브라우저 탐색, 요소 상호작용, 네트워크 요청 확인, 스크린샷 검증).
3. **API·비즈니스 로직은 꼼꼼히**: API 연동, 데이터 파싱/변환, N/1 계산, 상태 처리를 포함하는 태스크는 **정상 케이스 + 실패 케이스 + 엣지 케이스·경계값**을 모두 검증합니다.
4. **테스트 통과가 완료 조건**: 테스트를 통과해야만 태스크를 완료로 표시하고 다음 단계로 진행합니다.

### 테스트 검증 깊이 기준 (API·비즈니스 로직)

- **정상(Happy Path)**: 유효한 입력으로 기대 결과가 정확히 렌더링/반환되는지
- **실패**: API 오류(4xx/5xx), 타임아웃, 권한 없음, 데이터 없음 상태 처리
- **엣지·경계값**: 빈 목록, 정원 초과, 0/음수/최대 인원, 만료된 이벤트, 특수문자 입력
- **데이터 정합성**: N/1 정산 금액, 참여 인원 집계의 수치 정확성
- **사용자 플로우(E2E)**: 핵심 사용자 여정 전체를 처음부터 끝까지 검증

> 참고: UI 마크업(더미) 단계의 테스트는 렌더링·반응형·네비게이션 스모크 검증 위주로 수행하고, 데이터 정합성·실패/엣지 검증은 실데이터 연동 단계에서 집중적으로 수행합니다.

- [ ] TASK-031: Phase 2 통합 E2E 테스트 (Playwright MCP — 지출 입력 → N/1 계산 → 복사 → 공지 작성/읽기 → 독촉 복사)

### 완료 기준

- (UI) 정산·공지·독촉 화면이 더미 데이터로 완성되어 렌더링·반응형 검증 통과
- 항목별 지출 입력 시 N/1 금액이 정확히 계산되고 복사 가능
- 주최자 공지가 참여자 뷰에 노출되고 권한 분리가 동작
- 미응답자 대상 독촉 메시지가 정확히 생성·복사됨
- **Playwright MCP 테스트 통과** (정산은 비즈니스 로직이므로 꼼꼼히 검증):
  - 정상: 다항목 지출 합계 및 1인당 금액 정확성 검증
  - 실패: 지출 0건 정산 시도, 참여 인원 0명 나눗셈 방어 처리
  - 엣지·경계: 나누어떨어지지 않는 금액의 단수 처리, 0원/음수 입력 검증, 최대 인원 분배
  - 정합성: 항목 합계 = 각 항목 금액의 총합, 1인당 금액 × 인원 ≈ 총액 검증
  - 권한: 참여자가 공지 작성/지출 수정 시도 시 차단

### 완료 기준

- (UI) 카풀 화면이 더미 데이터로 완성되어 렌더링·반응형 검증 통과
- 드라이버 등록 → 탑승 신청 → 배정 확정의 전체 흐름이 동작
- 좌석 수를 초과하는 신청·확정이 차단됨
- **Playwright MCP 테스트 통과**:
  - 정상: 좌석 범위 내 신청·확정이 잔여 좌석에 정확히 반영
  - 실패: 좌석 만석 시 신청 차단, 미등록 드라이버 차량 신청 방어
  - 엣지·경계: 좌석 1석 차량 동시 신청 경합, 신청 취소 후 좌석 복구, 드라이버 본인 탑승 처리
  - 정합성: 확정 인원 ≤ 등록 좌석 수, 잔여 좌석 = 좌석 수 − 확정 인원

## 기술 부채 및 향후 고려사항

- **알림 자동화**: 현재 카카오톡/독촉은 수동 복사 방식. 향후 알림톡 API 또는 푸시 알림 연동 고려
- **정산 고도화**: 차등 분배(부분 참여, 항목별 참여자 지정), 송금 링크(토스/카카오페이) 연동
- **카풀 매칭**: 출발지 기반 자동 매칭, 지도 연동
- **이벤트 반복**: 정기 모임을 위한 반복 일정 생성
- **테스트 자동화**: Playwright E2E 테스트의 CI 파이프라인 통합 (현재는 Playwright MCP 수동 검증)
- **데이터 백업·마이그레이션**: Supabase 마이그레이션 버전 관리 체계 정립
- **국제화(i18n)**: 다국어 지원 (현재 한국어 단일)
- **접근성 심화**: WCAG 2.1 AA 준수 수준까지 확대
